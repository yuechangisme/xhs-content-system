# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## 项目概述

「懒人养生手册」小红书账号的内容生产基地。养生健康方向，面向 25-35 岁年轻人群，走「懒人友好、不折腾」的调性。

## 内容结构

```
content/
├── 公共素材库/               ← 所有已用图片汇总，新任务先来这里找图
├── render.js                 ← Puppeteer HTML→PNG 渲染脚本
├── 流程参考.md               ← 完整内容生产流程
├── 账号信息/                 ← 头像、背景图、品牌元素
│
├── YYYY-MM-DD-主题关键词/    ← 每篇内容一个独立文件夹
│   ├── 懒人养生手册-主题.html   ← 轮播图 HTML
│   ├── 主题-文案.md            ← 小红书发布文案
│   ├── images/                 ← 本篇用到的配图
│   ├── output/                 ← 导出的 PNG 文件
│   └── drafts/                 ← 过程文件
```

当用户需要为某篇帖子提供实拍图、橱窗商品图或截图素材时，Agent 应提前在该帖目录下创建 `images/user/`，并告知用户把图片放入该目录。建议按用途命名，例如 `p5-01-water-cup.jpg`、`p5-02-tea.jpg`、`p5-03-coffee.jpg`。正式 HTML 引用用户素材时优先使用 `images/user/` 中的文件，避免散落在聊天附件或临时目录导致后续对话无法复用。

## 关键命令

```bash
# 导出 HTML 为 PNG（3x 高清，1620x2160）
cd content
node render.js "YYYY-MM-DD-主题/懒人养生手册-主题.html" "文件前缀"
# 示例
node render.js "2026-05-15-高考饮食/懒人养生手册-高考饮食.html" "高考饮食"

# 本地预览 HTML（需要 HTTP 服务，否则 html2canvas 读不到本地图片）
npx http-server -p 8080 content/ --cors
# 打开 http://127.0.0.1:8080/ 访问
```

## 账号规则

- 标题 ≤ 20 字
- 标签 ≤ 10 个
- 品牌色：暖白底 #FEFAE0 + 健康绿 #3A7D44 + 暖橙 #F4A261
- 语气：朋友式分享，不端不装，口语短句
- 禁止词："本文将介绍""姐妹们""每天坚持""绝绝子"
- 人设："我也是踩坑过来的，帮你省时间"

## 视觉风格规范

生成或修改「懒人养生手册」小红书轮播图前，必须先阅读 `docs/STYLE_GUIDE.md`。

执行视觉任务时必须遵守其中的：

- 页面骨架：优先复用封面页、总览原则页、教程页、避坑对比页、方案清单页、总结收藏页。
- 字号和可读性规则：手机端优先，正文不能太小，关键文字优先保持 34px 以上。
- 文案规则：朋友式、懒人式、可执行，不写医学功效承诺，不写产品种草话术。
- 禁止项：不重新发明版式，不使用 `flex-1` / `spacer` / `space-evenly` 硬撑页面，不做低对比 CTA，不做文档式长段说明。

采用 V6.1 风格的新轮播正式内容，`manifest.json` 必须声明 `styleVersion: "lazy-health-v6.1"`，让 QA 使用 V6.1 分级字号规则。

如确需偏离 `docs/STYLE_GUIDE.md`，必须先说明原因并获得用户确认。

### 极简文字手册视觉方向

从 2026-06-04 起，后续「懒人养生手册」小红书轮播默认优先采用 `docs/STYLE_GUIDE.md` 中的「极简文字手册风格（V7 实验方向）」：

- 封面使用「大标题 + 一句副标题 + 主题相关生活物件手账插图 + 弱品牌 bar」。
- 配图必须和主题内容强相关，优先采用 AI 生成主题插图 + HTML 控制排版的混合方案。
- V7 封面插图必须在 AI 生图时就按固定封面图片窗口比例生成，默认约 `37:30` / `1.23:1`，并放入 `888×720` 显示窗口；禁止直接使用比例不稳定的原始生成图，也不要默认靠后期裁切补救。
- V7 正式封面必须使用 AI 生成的主题 PNG/JPG 插图，禁止用手写 SVG 当正式封面插图兜底。
- 内容页使用暖白底、轻分隔线、轻量编号、淡绿色高亮的手册式排版。
- 后续新帖默认做成 5 页版本：封面 1 页 + 内容 3 页 + 总结 1 页；如果用户确认需要商业承接、橱窗引导或真实生活图文补充，可以做 6 页版本，在总结页前增加 1 页图文场景/做法补充页。
- 内容页文字长度必须接近参考样例：每条是有场景和解释的 2-4 行说明，不写成过短口号清单，也不写成文档长段。
- 内容页正文必须停在底部品牌和页码上方，不能与页码或底部品牌重合；如果内容过长，优先压缩文案或拆分，不得缩小字号硬塞。
- 全篇编号最多写到 8 条；默认节奏是 P2/P3 各 3 条、P4 2 条加总结提醒，超过 8 条时合并相近项。
- 不使用拼音封面，不在封面塞大段解释，不做复杂信息图、卡片网格、图标堆砌或深绿默认内页。
- 用户确认文案内容好时，只调整排版，不擅自重写文案或添加判断句。

当前参考样例：`content/视觉实验/minimal-text-handbook-demo/懒人养生手册-极简文字手册样例.html`。

V7 新帖生成前必须逐项自查：

1. 页数是否为 5 或 6 页：5 页为 P1 封面、P2-P4 内容、P5 总结；6 页为 P1 封面、P2-P4 内容、P5 图文场景/做法补充、P6 总结。
2. 封面是否只有栏目名、大标题、一句副标题、AI 主题插图、底部品牌和页码；不得加卖点标签、解释卡或拼音。
3. 封面插图是否为 AI 生成 PNG/JPG，且生成时就按 `37:30` / `1.23:1` 窗口比例构图；不得用 SVG 兜底，不得用比例不稳定的原始图硬塞。
4. 内容页编号是否最多到 8；默认 P2/P3 各 3 条，P4 2 条加总结提醒。
5. 每条正文是否接近参考样例长度：有真实场景、原因或做法的 2-4 行说明；不得写成短口号清单。
6. 内容页正文是否全部停在页码和底部品牌上方，并留有安全距离。
7. 是否保持暖白底、轻分隔线、轻编号、淡绿色高亮；不得回到 V6.1 卡片网格、深绿内页或方案卡结构。

## 发布前目录边界

人工发布前检查前，正式内容必须先通过 `node pipeline.js promote "投稿内容/待制作/<dir>" --confirm-promote` 从 `待制作` 进入 `待投递`。

- `publish --dry-run` / `publish --confirm-publish` 只允许 `投稿内容/待投递/`。
- 如果平台发布成功但本地移动到 `已投递` 失败，不要重复 publish；应使用 `reconcile-move` 只修复本地归档。
- 不得手动修改 state.json 来伪造 promote、publish 或 reconciliation 状态。

## 自动发布停用规则

为降低平台自动化检测风险，从 v0.5.9 起，本项目不再执行小红书自动发布。

- 允许继续执行内容策划、正式草稿、render、QA、promote、发布前 dry-run 检查。
- `publish --dry-run` 仅作为人工发布前检查，不得理解为下一步自动发布准备。
- 禁止执行或建议执行 `publish --confirm-publish`；该入口应返回 `PUBLISH_DISABLED`。
- 禁止执行 scheduled publish 真发布分支；`schedule run-due --confirm-scheduled-publish --dry-run --task` 只能用于检查。
- 平台发布必须由用户人工在小红书完成。人工确认发布成功后，再按用户指令做本地归档或状态 reconciliation。
- 不得读取 cookie 或启动平台发布浏览器自动化来完成发布。

## 合集轮转规则

后续「懒人养生手册」发帖采用合集轮转制，详见 `docs/CONTENT_COLLECTION_ROTATION.md`。

当用户说「我们来发下一篇帖子」「做下一篇」「来一篇新的」「下一篇小红书」「今天发什么」等类似表达时：

1. 默认进入下一篇帖子策划流程。
2. 先按 `docs/CONTENT_COLLECTION_ROTATION.md` 判断这一篇轮到哪个合集。
3. 默认 10 篇循环为：懒人饮食手册 → 身体微习惯手册 → 打工人恢复手册 → 懒人食谱手册 → 懒人运动手册 → 懒人饮食手册 → 身体微习惯手册 → 打工人恢复手册 → 好物避坑手册 → 懒人食谱手册。
4. 如果存在 `docs/ACTIVE_CONTENT_PLAN.md` 中的短期活动计划，先判断是否需要穿插完成活动选题；活动完成后再回到普通合集轮转。
5. 先检查最近的 `待制作` / `待投递` / `已投递` 内容，尽量判断上一篇所属合集；无法可靠判断时，说明不确定，并默认从「懒人饮食手册」开始新一轮。
6. 自动提示本次轮到的合集、适合方向和 3 个候选选题；如活动优先，则提示本次属于活动穿插以及对应合集。
7. 如果用户直接指定明确选题，则优先尊重用户选题，只说明它归属哪个合集。
8. 不得手动修改 `state.json` 来记录或伪造轮转状态。

当用户问「下一个该做什么」「现在轮转到哪里」「下一篇做什么」等状态类问题时，回复必须包含：

1. 完整 10 篇轮转机制。
2. 当前判断所处的位置，以及判断依据（最近的 `待制作` / `待投递` / `已投递` 内容或短期活动计划）。
3. `docs/ACTIVE_CONTENT_PLAN.md` 中短期活动、临时提过但未制作的帖子；如果没有记录，明确说“暂无记录”。
4. 如果短期活动优先于普通轮转，必须说明“这次属于活动穿插，不代表普通轮转被永久改掉”。

当用户临时提到未来想做的帖子、活动、系列或选题，但没有立刻进入制作流程时，默认必须记录到 `docs/ACTIVE_CONTENT_PLAN.md` 的「临时提过但未制作」小节，不需要用户再次提醒。记录应包含：用户原始想法、归属合集或活动、当前状态、记录日期。这样后续其他发帖对话也能从项目文件中恢复这些计划。只有当用户明确说只是随便聊聊、不用记录，或想法含义无法判断时，才不立即写入并改为简短确认。

- 修改项目目录结构

- 修改运行方式

- 打 tag
- 发布版本

### 可以不更新的场景（需说明原因）

- 内部小重构
- 注释调整

- 是否更新 CHANGELOG：是/否

- 打 tag 但 CHANGELOG 不更新

## AGENTS.md Update Policy

AGENTS.md 不是普通变更记录，而是 Agent 的项目行为规则文件。只有当某条规则会影响未来 Agent 的长期行为时，才应该写入 AGENTS.md。

### 每次任务完成的报告必须包含

```
## Documentation Check
* README 是否需要更新：
* CHANGELOG 是否需要更新：
* CONTRACT 是否需要更新：
* AGENTS.md 是否需要更新：
* 原因：
```

### 必须考虑更新 AGENTS.md 的场景

1. **项目工作流程发生变化** — QA、render、publish、schedule、git、release 流程变化
2. **Agent 行为边界发生变化** — 新增禁止真实发布、必须 dry-run、必须确认等规则
3. **安全规则发生变化** — 禁止提交 cookie、config.local.js、账号信息、output PNG、本地路径等
4. **文档同步规则发生变化** — README / CHANGELOG / CONTRACT 的更新要求变化
5. **架构职责边界发生变化** — xhs-planner skill 与 xhs-content-system 的职责划分变化
6. **Agent 反复踩坑的问题被总结为长期规则** — 误写 PUBLISHED、忘记更新 README、误提交本地路径、跳过 QA 等

### 一般不需要更新 AGENTS.md 的场景

- 单次 bug 修复
- 单篇帖子内容修改
- 普通 CSS / 文案调整
- 不影响工作方式的内部代码重构
- 一次性的测试结果
- commit hash、tag 记录、临时日志

以上内容应放在 README、CHANGELOG、CONTRACT、commit message 或任务报告中，而不是 AGENTS.md。

本规则从现在开始长期生效。

## 定时发布确认模型（排期规则）

1. **确认发生在排期时，不是执行时。** 定时发布的确认由 `--confirm-schedule` 在 `schedule add` 时完成，执行时不再要求人在场。
2. **没有 `--confirm-schedule` 不得写入 confirmed schedule。** 未带此 flag 的 `schedule add` 只做预检查，不修改 state.json。
3. **未 confirmed 的排期永不自动发布。** `schedule.confirmed = false` 时，即使到时间也不触发发布。
4. **v0.3.0 禁止真实自动发布。** 当前版本只实现 schedule queue（add/list/cancel/status/due），不安装 node-schedule，不做常驻进程，不自动执行 publisher。
5. **已 PUBLISHED 的帖子禁止排期。** `schedule add` 前置检查会拒绝。
6. **QA 未通过的帖子禁止排期。** 只有 `QA_PASSED` 状态的帖子可以排期。
7. **已存在 active 排期的帖子禁止重复 add。** 每个帖子同时只能有一个 CONFIRMED 或 RUNNING 排期。

## Destructive Operation Safety Policy

### 核心原则

删除操作不可逆，必须逐级设防。任何删除、清理、移除文件的操作必须遵守以下规则。

### 规则

1. **先 audit，后删除。** 任何删除操作前必须先执行 dry-run audit，列出完整路径、文件数量、总大小、Git 跟踪状态、风险等级。

2. **未跟踪文件必须先备份。** 对于未被 Git 跟踪的文件（如 output/、state.json、error.log、调试临时文件），删除前必须先创建本地备份到 `cleanup-backup/`。备份完成后才允许执行删除。

3. **逐项确认，禁止批量。** 删除范围必须逐项确认，禁止用宽泛通配符（如 `*.png`、`temp-*`）批量删除。

4. **不删未列明的文件。** 不允许删除未在 audit 清单中明确列出的文件。

5. **删除后回归验证。** 删除后必须执行核心命令回归验证（status、qa、publish --dry-run、schedule due 等），确保管线不受影响。

6. **删除后明确区分提交类型。** 报告删除结果时，必须明确区分：
   - **Git 已提交变更** — 可通过 `git revert` 回滚
   - **工作区文件删除** — 未提交，不能通过 git 回滚
   - **未被 Git 跟踪的文件删除** — 无法通过 git 恢复，只能依赖备份

### 审计报告规范

7. **state.json 不是内容归档的唯一事实来源。** 已投递目录中的物理文件（HTML、manifest.json、output PNG）才是归档完整性的最终依据。state.json 仅记录进入 pipeline 管理后的运行状态。

8. **物理目录与 state 分别审计、分别描述。** 审计时必须分别检查已投递物理目录和 state.json 记录，并明确报告两者是否存在差异、差异原因、是否有风险。

9. **发现不一致时暂停开发。** 如果审计中发现数据不一致（如 PNG 数量存疑、state 与目录不匹配），必须先完成差异分析并确认无风险，才允许进入下一阶段开发。

10. **总结必须准确区分事实与推断。** 对于无法确认的历史操作（如被 gitignore 的文件的旧版本内容），必须标注为"无法确认"而非默认为"已删除"。不编造未经验证的数据。

## General Agent Coding Discipline

### 优先级声明

本节的通用纪律不替代项目安全规则。如有冲突，以本项目已定义的规则为准：

1. 发布安全规则（定时发布确认模型）
2. Destructive Operation Safety Policy
3. Documentation Update Policy
4. state / archive 边界规则
5. Scheduled Publish 确认规则

### 纪律

1. **先说明假设和风险再实现。** 当需求模糊、数据状态不确定、或变更可能产生副作用时，先列出你的理解和判断，让用户确认后再动手。不要默默选择一个路径。

2. **最小实现，不做未要求的功能。** 只实现任务明确要求的逻辑。不引入未要求的抽象层、配置项、灵活性。不预判未来需求。三行相似代码优于提前封装。

3. **手术式修改，不顺手重构。** 只改与任务直接相关的文件。不改相邻代码的格式、注释、命名。不改未损坏的逻辑。如果发现无关的死代码，可以提醒但不删除。每处改动必须能回溯到用户请求。

4. **每次任务必须有验收标准。** 实现前明确：怎么验证这个任务完成了？修改后运行对应的验证命令（status、qa、list、due、topic list 等），确认管线不受影响。

5. **发现不一致先暂停，不继续开发。** 审计中发现数据不一致、状态异常、或旧的假设被推翻时，先完成差异分析并确认无风险，才允许进入下一阶段。

6. **保持报告准确。** 区分事实与推断，区分已提交变更与工作区变更，不编造未验证的数据。如果不确定，直接说明不确定。
